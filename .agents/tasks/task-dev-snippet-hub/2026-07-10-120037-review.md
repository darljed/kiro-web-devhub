# Dev Snippet & Content Hub with portfolio page and dark mode

This change replaces the Next.js starter page with a static Dev Snippet Hub: a portfolio home page, a filterable snippet index, and detail pages for four mock code snippets. Dark mode is wired through next-themes with class-based Tailwind v4 toggling, and client components are isolated to the three interactive elements (theme toggle, search, copy button). All routes are statically generated at build time via `generateStaticParams` with `dynamicParams = false`.

Watch for: theme toggle flash/mismatch on first render due to `useTheme` reading `theme` directly instead of `resolvedTheme` (confirmed); the search input lacks an accessible label (confirmed); the CopyButton has no error handling for clipboard API rejection (confirmed).

## High-level view

The Navbar is a full client component because it hosts the theme toggle, which means the navigation links, the logo, and all their markup are excluded from the server render tree. This is a minor SSR trade-off for a small nav, but worth noting for future growth.

Dark mode relies on a `@custom-variant dark (&:where(.dark, .dark *))` declaration in globals.css paired with next-themes setting the `class` attribute on `<html>`. The Navbar's toggle logic reads `theme` (which can be `"system"`) rather than `resolvedTheme`, so the first toggle after page load when the theme is `"system"` will flip to `"light"` regardless of the user's OS preference.

The search input in SnippetSearch lacks a `<label>` or `aria-label`, which is an accessibility gap for screen reader users relying on landmark or form-field navigation.

<summary>Issues (4)</summary>

1. **Theme toggle reads `theme` instead of `resolvedTheme`** — When the active theme is `"system"`, clicking the toggle switches to `"light"` unconditionally. Use `resolvedTheme` from `useTheme()` to determine the current effective theme before toggling.
2. **Search input missing accessible label** — The `<input>` in SnippetSearch has a `placeholder` but no `<label>`, `aria-label`, or `aria-labelledby`. Add `aria-label="Search snippets"` to the input.
3. **CopyButton has no error handling for clipboard API** — `navigator.clipboard.writeText` can throw (permissions denied, insecure context). The `await` will produce an unhandled rejection. Wrap in try/catch or add a `.catch()`.
4. **Footer uses `new Date().getFullYear()` at render time in a server component** — This evaluates at build time for a static site, so the year will be stale if the site is rebuilt infrequently. Acceptable for a static portfolio, but worth noting.

## Theme toggle behavior on `"system"` default

The Navbar reads `theme` from `useTheme()` and toggles with `setTheme(theme === "dark" ? "light" : "dark")`. When the page first loads with `defaultTheme="system"`, `theme` is the string `"system"` — not `"dark"` or `"light"`. The conditional evaluates `"system" === "dark"` as false and sets `"dark"`, which happens to be correct if the user's OS is in light mode, but sets `"dark"` even when the OS is already dark (so the user sees no visible change on first click). The fix is to use `resolvedTheme`:

```tsx
const { resolvedTheme, setTheme } = useTheme();
// ...
onClick={() => setTheme(resolvedTheme === "dark" ? "light" : "dark")}
```

`resolvedTheme` is undefined on the server, so the icon toggle using CSS `dark:block` / `dark:hidden` correctly avoids hydration mismatch — only the `onClick` handler needs the switch to `resolvedTheme`. Confidence: **confirmed** — traceable in Navbar.tsx line 9 and line 35.

## Accessibility gap in search input

The `<input>` in `SnippetSearch.tsx` has `placeholder="Search snippets..."` but no programmatic label. Placeholder text disappears on focus and is not reliably announced by all screen readers as a label. Adding `aria-label="Search snippets"` is a one-line fix. Confidence: **confirmed** — line 25 of SnippetSearch.tsx.

## CopyButton clipboard error path

`navigator.clipboard.writeText` returns a Promise that can reject if the Permissions API denies access or if the page is in an insecure context (HTTP without localhost). The current code `await`s the call without try/catch:

```tsx
const handleCopy = async () => {
  await navigator.clipboard.writeText(code); // unhandled rejection on failure
  setCopied(true);
  // ...
};
```

The function should wrap in try/catch with a user-visible error state or at minimum swallow the rejection gracefully. Confidence: **confirmed** — CopyButton.tsx line 10.

## Unnamed section landmark on home page

The home page `<section>` has no `aria-labelledby` or `aria-label`, and the `<h1>` contains a person's name rather than a page title. Screen readers navigating by landmark will announce the section without a descriptive name. Low-severity for a portfolio site but worth an `aria-label="Introduction"` or similar. Confidence: **possible** — depends on screen reader behavior with unnamed sections.
