# Dev Snippet & Content Hub (v2)

This change builds a static Next.js 16 Dev Snippet Hub with a portfolio home page, filterable snippet index, and detail pages with copy-to-clipboard. Dark mode is managed via next-themes with class-based Tailwind v4 toggling. Client components are isolated to three interactive concerns (theme toggle, search, copy). All routes are statically generated at build time. The three issues from the v1 review — `resolvedTheme` usage, search input accessibility, clipboard error handling — have all been addressed.

Watch for: CopyButton silently swallows clipboard failures with no user feedback (confirmed); the Navbar's Sun/Moon icon rendering relies on CSS visibility rather than conditional React rendering, which ships both SVGs to the DOM on every page load (confirmed, low severity).

## High-level view

The v1 fixes are correctly applied. Navbar now uses `resolvedTheme` for toggle logic, the search input carries `aria-label="Search snippets"`, and CopyButton wraps the clipboard call in try/catch.

The CopyButton's error path catches the exception but provides no feedback to the user — the button remains in its default state as if nothing happened. A user on an insecure context (HTTP without localhost) will click "Copy" repeatedly with no indication that it's failing.

## CopyButton error UX gap

The catch block is empty:

```tsx
const handleCopy = async () => {
  try {
    await navigator.clipboard.writeText(code);
    setCopied(true);
    setTimeout(() => setCopied(false), 2000);
  } catch {
    // Clipboard API may fail in insecure contexts or when permission is denied
  }
};
```

When the clipboard write fails, `setCopied(true)` is never reached, so the button stays in its default "Copy" state. The user has no way to distinguish "I didn't click hard enough" from "this can never work in my context." A minimal improvement would be a brief "Copy failed" state using the same 2-second timeout pattern already in place. Confidence: **confirmed**.

## Navbar icon rendering strategy

The theme toggle renders both icons and uses Tailwind's `dark:block` / `dark:hidden` to show the correct one:

```tsx
<Sun className="hidden h-5 w-5 dark:block" />
<Moon className="block h-5 w-5 dark:hidden" />
```

This avoids hydration mismatch with next-themes (since `resolvedTheme` is undefined on the server, a conditional render would produce a client/server divergence). The trade-off is both SVGs are always in the DOM — negligible for two small icons, but worth noting if the nav grows more complex. Confidence: **confirmed**.
