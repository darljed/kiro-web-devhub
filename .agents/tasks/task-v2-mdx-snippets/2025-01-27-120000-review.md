# MDX snippet system with build-time Shiki highlighting

This change replaces the hardcoded `lib/data.ts` snippet store with a file-based MDX content system. Each snippet lives in `content/snippets/*.mdx` with YAML frontmatter and rich prose content. Syntax highlighting happens at build time via rehype-pretty-code (Shiki), eliminating client-side JS for code rendering. The snippet detail page gains multi-tab code blocks, a "Copy as AI Prompt" button, tag-based filtering, and auto-generated OG images. Twelve production-quality snippets ship across Splunk, React/Next.js, Python, and Bash categories.

**Watch for:** Missing `@tailwindcss/typography` plugin makes the `prose`/`prose-invert` classes on the MDX content container non-functional — all rendered markdown prose (headings, lists, paragraphs) falls back to browser defaults (confirmed). The `CopyAsPrompt` and `CopyButton` components silently swallow clipboard failures with no user feedback (confirmed). The `--webpack` build flag is a Turbopack workaround that should be documented as intentional rather than left as an unexplained deviation (confirmed).

## High-level view

The content architecture uses a dual-export pattern: MDX files export a `codeBlocks` named constant alongside their default component, separating machine-readable code (for copy buttons, tabs, and prompt generation) from the human-readable prose rendered by the MDX pipeline. This keeps data flow simple but creates a maintenance burden — every code block exists twice in each MDX file (once in the `codeBlocks` export, once in the rendered markdown), and drift between them would cause the copy button to serve stale code.

The Shiki integration uses a single dark theme (`github-dark`) with `keepBackground: false`, meaning code blocks always render with the dark palette regardless of the site's active theme. The `CodeTabs` component renders the `codeBlocks` strings as plain text (no Shiki highlighting), while the same code in the MDX prose section below gets full syntax highlighting — a visible quality gap between the two presentation paths.

The typography gap is the most impactful functional issue: the snippet detail page wraps MDX output in `prose prose-zinc dark:prose-invert` classes, but without `@tailwindcss/typography` installed, these classes resolve to nothing. The manual `.prose pre` overrides in `globals.css` are equally inert without the plugin providing the base `.prose` styles.

No `metadataBase` is configured in the app, so Next.js defaults OG image URLs to `http://localhost:3000`. Deployed OG images may not resolve correctly depending on platform URL inference.

<details>
<summary>Issues (5)</summary>

1. **Missing @tailwindcss/typography plugin** — Install `@tailwindcss/typography` and import it in `globals.css` (Tailwind v4 pattern: `@import "@tailwindcss/typography"`). Without it, the `prose`/`prose-invert` classes on the MDX container are dead CSS.
2. **Clipboard failure is invisible** — Both `CopyButton` and `CopyAsPrompt` catch clipboard errors with an empty block. Add a brief visual indicator (e.g., "Failed" state or tooltip) so users know the copy didn't succeed.
3. **Duplicated code in MDX files** — Every snippet's code appears once in the `codeBlocks` export and again in the markdown body. A remark/rehype plugin or build-time script could extract code blocks from the MDX AST and auto-generate the export, eliminating the duplication.
4. **Single-theme Shiki highlighting** — The `github-dark` theme means code blocks are always dark. If adaptive theming is desired later, a dual-theme setup (one light, one dark) with CSS-variable switching would be the path forward.
5. **No metadataBase set** — Next.js warns about resolving relative OG image URLs without `metadataBase`. Set it in `app/layout.tsx` metadata export to the production domain to ensure OG images resolve correctly when shared.

## Content architecture: MDX with dual exports

The snippet detail page dynamically imports each MDX module at build time:

```typescript
const mdxModule = await import(`@/content/snippets/${id}.mdx`);
const MdxContent = mdxModule.default;
const codeBlocks = mdxModule.codeBlocks ?? {};
```

If someone edits the fenced code block in the prose without updating the `codeBlocks` export (or vice versa), the copy button silently serves stale code. For a 12-snippet library this is manageable. At 50+ snippets it would warrant a build-time extraction step — a remark plugin that walks the AST for fenced code blocks and generates the `codeBlocks` export automatically.

## Typography plugin gap

The snippet detail page renders MDX content inside:

```tsx
<div className="prose prose-zinc mt-10 max-w-none dark:prose-invert prose-pre:p-0 prose-pre:bg-transparent prose-pre:border-0">
  <MdxContent />
</div>
```

`@tailwindcss/typography` is not in `package.json` and not imported in `globals.css`. In Tailwind v4, typography requires both installing the package and adding `@import "@tailwindcss/typography"` to the CSS entry. The `prose-pre:*` modifiers are also typography-plugin utilities — equally inert.

The `globals.css` overrides for `.prose pre` and `.prose code::before/::after` target a selector that provides no base styles. These overrides only become meaningful once the plugin is installed.

## Clipboard error handling

Both clipboard-using components follow the same pattern:

```typescript
} catch {
  // Clipboard API may fail in insecure contexts or when permission is denied
}
```

In HTTP contexts or when the Permissions API denies access, the button does nothing visible. A transient "Failed to copy" state (matching the existing "Copied!" pattern) would eliminate the confusion.

## Build tooling: Turbopack bypass

The build script changed from `next build` to `next build --webpack` because Turbopack cannot serialize rehype plugin functions. This is documented in the task findings but not in the project source. A future developer removing the flag would get a cryptic serialization error. A comment in `next.config.mjs` explaining the constraint would save debugging time.

## CodeTabs: unhighlighted code in tabs

`CodeTabs` renders `codeBlocks` values as plain text:

```tsx
<code className="text-sm leading-relaxed text-zinc-100">
  {activeTab.code}
</code>
```

The tabbed code view has no syntax highlighting — only the MDX prose section below gets Shiki treatment. Users who interact primarily with the tabs (the more prominent UI element, appearing above the prose) see visually inferior code. This is an inherent limitation of the architecture: Shiki runs in the rehype pipeline at build time, and CodeTabs is a client component that receives raw strings. A solution would require either pre-highlighting the `codeBlocks` strings during build (custom webpack loader) or using a lightweight client-side highlighter for the tabs.
