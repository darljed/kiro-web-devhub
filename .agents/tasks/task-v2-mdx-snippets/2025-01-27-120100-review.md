# MDX snippet system with build-time Shiki highlighting (v2)

This change replaces the hardcoded `lib/data.ts` snippet store with a file-based MDX content system. Each snippet lives in `content/snippets/*.mdx` with YAML frontmatter and rich prose content. Syntax highlighting happens at build time via rehype-pretty-code (Shiki), eliminating client-side JS for code rendering. The snippet detail page gains multi-tab code blocks, a "Copy as AI Prompt" button, tag-based filtering, and auto-generated OG images. Twelve production-quality snippets ship across Splunk, React/Next.js, Python, and Bash categories.

The four fixable issues from the first review (typography plugin, metadataBase, webpack comment, clipboard failure feedback) have all been addressed. The two accepted trade-offs (duplicated code in MDX files, single dark theme) remain as intentional decisions.

**Watch for:** The `CodeTabs` component renders raw strings without syntax highlighting while the MDX prose section below gets full Shiki treatment — a visible quality gap between the two code presentation paths (confirmed). The `metadataBase` uses a placeholder domain (`devhub.example.com`) that will need updating before production deployment (confirmed).

**Verdict**: APPROVED

## High-level view

The content architecture works as designed: MDX files export both a rendered component (for prose with Shiki-highlighted code blocks) and a `codeBlocks` constant (for copy buttons, tabs, and prompt generation). The dual-export pattern creates a maintenance burden with duplicated code, but at 12 snippets it's manageable and was accepted as an intentional trade-off.

The remaining architectural gap is the CodeTabs component: it renders `codeBlocks` strings as plain text because Shiki runs at build time in the rehype pipeline, while CodeTabs is a client component receiving raw strings. The tabbed code view (the more prominent UI element, appearing above the prose) has visually inferior rendering compared to the MDX body below it.

## CodeTabs: unhighlighted code in tabs

The tabbed code view renders `codeBlocks` values as plain text:

```tsx
<pre className="overflow-x-auto rounded-b-lg border border-zinc-200 bg-zinc-950 p-4 dark:border-zinc-700">
  <code className="text-sm leading-relaxed text-zinc-100">
    {activeTab.code}
  </code>
</pre>
```

Users who interact primarily with the tabs see white-on-dark monospace text with no language-aware coloring. The same code rendered via the MDX prose section below gets full Shiki treatment (keyword colors, string highlighting, etc.). This asymmetry is an inherent limitation of the architecture — Shiki processes markdown code fences during the rehype pass at build time, but CodeTabs is a client component that receives raw strings at runtime.

Paths to fix:
- A custom webpack loader that imports each MDX's `codeBlocks`, runs Shiki's `codeToHtml()` on them, and exports pre-rendered HTML strings alongside the raw strings
- A build-time script that generates a `codeBlocks.highlighted.ts` file per snippet
- A lightweight client-side highlighter (e.g., Shiki's WASM bundle) loaded on-demand in CodeTabs — at the cost of adding client JS

None of these are urgent for a 12-snippet site where the prose section provides the highlighted version, but the quality gap is visible.
