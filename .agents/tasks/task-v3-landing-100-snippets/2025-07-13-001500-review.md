# Landing page redesign and 100-snippet content expansion

The landing page has been rewritten from a generic "code snippet vault" pitch to an AI-era value proposition: curated production patterns that AI assistants can't generate from scratch. The old two-column hero with illustration is replaced by a centered headline and new sections (LivePreview, Categories). One hundred new MDX snippet files are added across 12 categories, taking the total from 22 to 122. The build and lint pass cleanly.

**Watch for:** Clipboard error handling gap in LivePreviewSection (confirmed). Three dead component files left behind (confirmed). Category count distribution deviates slightly from the specification — Redis and GraphQL each got 2 instead of 3 (confirmed). Hardcoded category counts in CategoriesSection will drift as content is added (likely).

## High-level view

The hero section shifts from a split layout with an SVG illustration to a single centered column with a gradient headline and a CTA pointing to the snippet browser. The anchor changes from `#use-cases` to `#categories`, matching the new section. `HeroIllustration.tsx`, `UseCasesSection.tsx`, and `FeatureIllustrations.tsx` become dead code — still in the tree but no longer imported.

LivePreviewSection renders a hardcoded `useDebounce` snippet card with a copy button. The code is a template literal duplicated from the actual snippet, not loaded from the content layer, so it won't stay in sync. The copy handler lacks the error handling every other clipboard interaction in the codebase has.

CategoriesSection renders a static array of 14 category objects with per-category snippet counts. The counts are accurate today but will silently drift as content changes. The content layer already parses all MDX at build time — deriving these counts would cost nothing.

The 100 new MDX files follow the established pattern. The distribution is close to but not exactly what was specified: TypeScript got 18 (requested 15), Redis and GraphQL each got 2 (requested 3 each).

## Clipboard handling inconsistency in LivePreviewSection

The existing `CopyButton` and `CopyAsPrompt` components both use async/await with try/catch around `navigator.clipboard.writeText`, transitioning to a "failed" state on error. LivePreviewSection calls it synchronously without catching the returned Promise rejection:

```typescript
const handleCopy = () => {
  navigator.clipboard.writeText(sampleCode);
  setCopied(true);
  setTimeout(() => setCopied(false), 2000);
};
```

In insecure contexts (HTTP, some iframes) or when clipboard permission is denied, this throws an unhandled rejection while still showing the success checkmark. Adopt the same async/try/catch/failed-state pattern used in `CopyButton.tsx`.

## Dead code: three orphaned component files

The diff removes imports but leaves the source files:

- `components/HeroIllustration.tsx`
- `components/landing/UseCasesSection.tsx`
- `components/FeatureIllustrations.tsx`

No runtime impact (tree-shaken), but dead code in the source tree.

## Static category data vs. dynamic content

CategoriesSection defines counts as literal numbers:

```typescript
{ name: "TypeScript", count: 18, ... },
{ name: "Redis", count: 3, ... },
```

The content layer (`lib/snippets.ts`) already reads and parses all MDX files at build time. Deriving category counts from that source would eliminate the drift risk without any runtime cost since the site is statically generated. The same concern applies to StatsSection's "122+" value, though the "+" suffix gives it more shelf life.
