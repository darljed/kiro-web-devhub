# Landing page redesign and 101-snippet content expansion (v2)

The v1 review raised five issues. Three are resolved: clipboard error handling in LivePreviewSection now uses async/await with try/catch and a "failed" state; the three dead component files (HeroIllustration, UseCasesSection, FeatureIllustrations) are deleted; GraphQL now has 3 snippets. Two issues persist: category counts remain hardcoded, and the preview snippet is still a duplicated string literal. StatsSection was updated to "120+" which accurately reflects 123 snippets with shelf-life headroom.

**Watch for:** Snippet count is 101, not the specified 100 (confirmed). Per-category distribution still deviates from spec in several categories (confirmed). Hardcoded category counts will drift on next content change (likely).

**Verdict**: APPROVED

## High-level view

The 101 new MDX files bring the total to 123. The per-category distribution deviates from the specification (TypeScript got 18 vs 15 requested, Redis only added 2 new, SQL got 5 vs 6, Terraform got 6 vs 7) but the total is within one of the target. The CategoriesSection counts are hardcoded rather than derived from the content layer, which means they'll drift silently on the next content addition.

Any future snippet addition requires manually updating both the content directory and the static array in `CategoriesSection.tsx`. The content layer (`lib/snippets.ts` → `getAllSnippets()`) already reads all MDX and exposes `category` — a server component or build-time helper could derive these counts with zero runtime cost.

## Hardcoded preview snippet

LivePreviewSection embeds a `useDebounce` hook as a template literal rather than loading from the actual MDX file (`content/snippets/react-use-debounce.mdx` presumably contains identical code). If that snippet is ever updated, the landing page preview diverges silently. For a static marketing section this is low-severity, but future maintainers won't know to update both locations.
