## Overview

A thread-safe token bucket rate limiter for Rust async applications using Tokio. The token bucket algorithm allows bursts up to the bucket capacity while maintaining a long-term average rate, making it ideal for API clients, request throttling, and resource access control.

Key design decisions:
- **Arc + Mutex** enables safe sharing across tasks and threads
- **Lazy refill** calculates tokens on demand instead of running a background timer
- **Configurable burst capacity** separates sustained rate from burst allowance
- **Timeout support** prevents indefinite blocking when the system is overloaded
- **Clone-friendly** via Arc for easy sharing in concurrent handlers

    pub max_tokens: u64,

    tokens: f64,
    max_tokens: u64,
    refill_rate: u64,
    last_refill: Instant,
}

impl TokenBucketInner {
    fn refill(&mut self) {
        let now = Instant::now();

        let new_tokens = elapsed * self.refill_rate as f64;

        self.tokens = (self.tokens + new_tokens).min(self.max_tokens as f64);
        self.last_refill = now;
    }

    fn try_acquire(&mut self, tokens: u64) -> bool {
        self.refill();
        if self.tokens >= tokens as f64 {
            self.tokens -= tokens as f64;
            true
        } else {
            false
        }
    }

    fn tokens_available(&mut self) -> u64 {
        self.refill();
        self.tokens as u64
    }

    fn time_until_available(&mut self, tokens: u64) -> Duration {
        self.refill();
        if self.tokens >= tokens as f64 {
            return Duration::ZERO;
        }
        let deficit = tokens as f64 - self.tokens;
        let seconds = deficit / self.refill_rate as f64;
        Duration::from_secs_f64(seconds)
    }
}

#[derive(Debug, Clone)]
pub struct RateLimiter {
    inner: Arc<Mutex<TokenBucketInner>>,
}

impl RateLimiter {
    pub fn new(config: RateLimiterConfig) -> Self {
        let inner = TokenBucketInner {
            tokens: config.max_tokens as f64,
            max_tokens: config.max_tokens,
            refill_rate: config.refill_rate,
            last_refill: Instant::now(),
        };

        Self {
            inner: Arc::new(Mutex::new(inner)),
        }
    }

    pub async fn try_acquire(&self, tokens: u64) -> bool {
        let mut inner = self.inner.lock().await;
        inner.try_acquire(tokens)
    }

    pub async fn acquire(&self, tokens: u64) {
        loop {
            let wait_time = {
                let mut inner = self.inner.lock().await;
                if inner.try_acquire(tokens) {
                    return;
                }
                inner.time_until_available(tokens)
            };
            tokio::time::sleep(wait_time).await;
        }
    }

    pub async fn acquire_with_timeout(
        &self,
        tokens: u64,
        timeout: Duration,
    ) -> Result<(), RateLimitError> {
        let deadline = Instant::now() + timeout;
        loop {
            let wait_time = {
                let mut inner = self.inner.lock().await;
                if inner.try_acquire(tokens) {
                    return Ok(());
                }
                inner.time_until_available(tokens)
            };
            if Instant::now() + wait_time > deadline {
                return Err(RateLimitError::Timeout);
            }
            tokio::time::sleep(wait_time).await;
        }
    }

    pub async fn available(&self) -> u64 {
        let mut inner = self.inner.lock().await;
        inner.tokens_available()
    }
}

#[derive(Debug, thiserror::Error)]
pub enum RateLimitError {
    #[error("rate limit acquisition timed out")]
    Timeout,
}

## Notes

- For very high throughput, consider a sharded approach with multiple buckets to reduce contention
- The `thiserror` crate is used for ergonomic error types; replace with manual `impl` if you prefer no dependencies
- Token bucket allows full burst up to `max_tokens` followed by sustained `refill_rate` throughput
- For distributed rate limiting, combine with Redis (see the Redlock pattern) for cross-process coordination
- Consider using `tokio::sync::Semaphore` for simpler concurrency limiting without the time-based refill
- Add metrics (prometheus counters) for tokens granted vs denied to monitor throttling in production
