## Overview

The cache-aside pattern (lazy loading) checks Redis before the database, caching results on miss. This reduces database load for read-heavy workloads while keeping implementation simple and cache consistent with source data.

## Code

```
GET key           -> cache hit? return
SET key value EX 3600  -> cache miss: store with TTL
DEL key           -> on write: invalidate
```

## Notes

- TTL prevents stale data from persisting indefinitely; choose based on acceptable staleness
- Cache stampede occurs when many processes simultaneously discover an expired key and hit the DB
- `SET NX` lock ensures only one process rebuilds while others wait (mutex pattern)
- `MGET` reduces round trips by batching multiple key lookups in one command
- Hash (`HSET/HGET`) enables partial field updates without invalidating the entire object
- Write-through alternative: update cache on every write (fresher but more complex)
