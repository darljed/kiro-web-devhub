## Overview

A Terraform module pattern for managing infrastructure across multiple environments (dev, staging, production) while maintaining DRY principles and preventing state conflicts. This approach uses separate environment directories with shared modules, enabling teams to deploy independently to different environments.

Key design decisions:
- **Input validation** prevents invalid environment names from causing misconfigurations
- **Dynamic subnet calculation** from the VPC CIDR avoids manual IP math
- **Cost optimization** via `single_nat_gateway` flag for non-production environments
- **Remote state isolation** with per-environment S3 keys and DynamoDB locking
- **Consistent tagging** through `common_tags` local and provider-level `default_tags`

## Notes

- Use `terraform workspace` only if your team agrees on the workflow; separate directories are often clearer
- Always use DynamoDB locking to prevent concurrent state modifications
- Pin provider versions in modules to avoid breaking changes on `terraform init`
- Consider using `terragrunt` for additional DRY patterns across environments
- Add `lifecycle { prevent_destroy = true }` to critical resources like VPCs in production
- Use `terraform plan -var-file=env.tfvars` as an alternative to separate directories for simpler setups
- Enable S3 bucket versioning on the state bucket for recovery from corrupted state
