## Overview

A production-grade Kubernetes deployment configuration that ensures zero downtime during updates. This manifest combines rolling update strategy, health probes, pod disruption budgets, and horizontal autoscaling to maintain availability throughout deployments, node drains, and traffic spikes.

Key design decisions:
- **maxUnavailable: 0** ensures the old pod stays running until the new one passes readiness checks
- **preStop hook** with a sleep gives the Service time to remove the pod from endpoints before termination
- **Startup probe** separates slow-starting apps from unhealthy apps, preventing premature restarts
- **Topology spread** distributes pods across availability zones for fault tolerance
- **PodDisruptionBudget** protects against cluster operations taking down too many pods simultaneously

## Notes

- Set `terminationGracePeriodSeconds` greater than your longest request timeout plus the preStop sleep
- The preStop sleep (10s) must be shorter than `terminationGracePeriodSeconds` (60s) to allow graceful shutdown
- Use `maxUnavailable: 0` with `maxSurge: 1` for safest rollouts; increase `maxSurge` for faster deployments
- Separate liveness from readiness probes: liveness should only check "is the process alive", readiness should check dependencies
- Set resource requests equal to limits for Guaranteed QoS class on critical workloads
- The HPA `scaleDown` stabilization window (300s) prevents flapping during traffic fluctuations
- Consider using `topologySpreadConstraints` over pod anti-affinity for more flexible scheduling
